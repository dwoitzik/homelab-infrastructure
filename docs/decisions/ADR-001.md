# ADR-001: Unbound as recursive DNS resolver

**Date:**
**Status:** Accepted

## Context

The homelab requires a local DNS resolver for all internal clients. The options considered were:

- Rely on a public upstream (1.1.1.1, 8.8.8.8)
- Use a forwarding resolver (Pi-hole forwarding to Cloudflare/Quad9)
- Run a full recursive resolver (Unbound querying root servers directly)

## Decision

Deploy Unbound as a local recursive resolver, placed behind AdGuard Home in the DNS chain:

```
Client → AdGuard Home (filtering) → Unbound (recursive) → Root DNS servers
```

## Reasons

**Privacy:** Forwarding resolvers send every DNS query to a third-party provider. With Unbound, queries are distributed across authoritative servers — no single provider sees the full query history.

**No upstream dependency:** The lab operates independently of external DNS providers. If Cloudflare or Google DNS have an outage, internal resolution continues unaffected.

**Performance:** Unbound caches responses locally and resolves subsequent queries instantly. Kernel network buffers are tuned via Ansible (`net.core.rmem_max`, `net.core.wmem_max`) to handle high-volume UDP traffic.

**Learning value:** Operating a recursive resolver requires understanding the DNS hierarchy, TTLs, and DNSSEC — directly relevant to production infrastructure roles.

## Trade-offs

- First-query latency is higher than forwarding (full recursion vs. cached upstream)
- DNSSEC validation must be maintained manually
- More operational complexity than a simple forwarder

## Consequences

Unbound is managed via Ansible and runs as a systemd service on both Raspberry Pi edge nodes. AdGuard Home is configured to use `127.0.0.1:5335` (Unbound) as its sole upstream.
